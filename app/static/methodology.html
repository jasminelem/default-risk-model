Methodology

A detailed overview of the data sources, machine learning pipeline, explainability framework, CDS spread analysis, peer benchmarking, and backtesting calibration behind this platform.

1. Data Pipeline & Sources

The foundation of this model is a clean, time-series dataset constructed from multiple financial databases. The process ensures no data leakage and respects the temporal nature of financial reporting.

DATA SOURCES (WRDS)
  • Compustat Fundamentals (funda/fundq): Annual and quarterly financial statements (balance sheet, income statement).
  • CRSP Stock Prices (dsf): Daily stock returns and prices used to calculate market volatility and momentum.
  • FJC Bankruptcy Database: Historical records of corporate bankruptcy filings used to define the target variable.
  • Macroeconomic Data (FRED): Interest rates (Fed Funds, Treasury yields) and credit spreads (Baa/Aaa) as external risk factors.

2. Model Architecture

A single unified discrete-time hazard model (XGBoost) predicts default probability at both the 12-month and 5-year horizons. Instead of two separate models, the prediction horizon is encoded as a feature (horizon_months = 12 or 60). Each company is scored twice to produce both PDs from one model.

Monotonicity Guarantee
XGBoost monotone_constraints enforce that P(default) is non-decreasing in horizon_months. The 5-year PD is always ≥ the 12-month PD by construction.
Calibration
CalibratedClassifierCV with 5-fold isotonic regression maps raw scores to true default probabilities using ~400K cross-validated predictions.
The model uses 60+ features (leverage, profitability, liquidity, market signals, macro conditions) plus the horizon feature. Hyperparameters are tuned via Optuna (40 trials) on a time-based validation set.

3. Explainability (SHAP)

To move beyond a "black box," we use SHAP (SHapley Additive exPlanations) to interpret model outputs.

Global View: We generate summary plots (beeswarm/bar) to show which features are most important for the model overall and in which direction they push the prediction.

Local View: For every individual company prediction, we calculate the exact contribution (SHAP value) of every feature. A positive value pushes the prediction towards "Default," a negative value pushes it towards "No Default."

The dashboard shows local SHAP values for each company at both horizons. Since a single model is used, the feature attributions are consistent across horizons — differences reflect how the model weights features differently at 12 months vs 5 years.

4. CDS Spread Analysis & Fair Value

The platform bridges fundamental credit analysis with market pricing by translating between default probabilities and CDS spreads using the standard reduced-form credit pricing framework.

Fair Value CDS Spread
For every company, the model PD is converted into an implied "fair" CDS spread:
spread = −ln(1 − PD) × (1 − R) × 10,000
Where R = 40% (standard recovery assumption for senior unsecured debt). The result is the annualised spread in basis points that a CDS contract should trade at if the model's PD is correct.
Market-Implied PD
Users can enter an observed CDS spread to reverse-engineer the market's implied default probability:
PD = 1 − exp(−spread / (1 − R) / 10,000)
This is the inverse of the fair-spread formula. The recovery rate can be adjusted for seniority: 60% for senior secured, 40% for senior unsecured, 25% for subordinated.
RICH / CHEAP RELATIVE VALUE FLAG
When both model PD and market-implied PD are available, the system flags divergences using a simple ratio test:
Condition Flag Interpretation
Market PD > 2× Model PD CHEAP Market overprices default risk relative to fundamentals. The bond or CDS may offer value.
Market PD < 0.5× Model PD RICH Market underprices default risk. The bond may be overvalued relative to the fundamental model view.
0.5× ≤ ratio ≤ 2× FAIR Market pricing is broadly consistent with the model's fundamental assessment.
Assumptions & Limitations: The reduced-form model assumes a constant hazard rate and a fixed recovery rate. Real CDS pricing involves term structure, counterparty risk, and liquidity premia not captured here. The fair spread should be treated as an indicative benchmark, not a trading signal.

5. Peer Group Comparison

Each company's default probability is contextualised against its industry peers. This helps distinguish between companies that are risky in absolute terms versus companies that are risky relative to their sector.

Industry Classification: Companies are grouped by SIC sector using Compustat's historical SIC codes, mapped to 10 broad industry sectors (Manufacturing, Services, Retail, etc.). Financial firms (SIC 6000-6999) and utilities (SIC 4900-4999) are excluded from the model universe due to fundamentally different capital structures.

Percentile Ranking: For each company, we compute its percentile rank within its sector by comparing its calibrated 12-month PD against all other companies in the same industry. A firm at the 90th percentile has a higher predicted default risk than 90% of its sector peers.

Peer Selection: The five closest peers by default probability are displayed alongside the target company, enabling direct comparisons between firms with similar risk profiles.

PEER STATISTICS PROVIDED
Metric Description
Risk Percentile Where the company's PD falls relative to all firms in its sector (0th = lowest risk, 100th = highest risk).
Peer Median PD The median default probability across all companies in the same industry.
Peer Min / Max PD The range of default probabilities within the sector, showing the best and worst positioned firms.
Closest Peers The five companies with the most similar default probability, enabling like-for-like fundamental comparison.
All PDs for peer comparison are pre-computed at startup using the unified model at horizon=12, ensuring consistency with the company-level predictions.

6. Backtesting & Calibration

To validate that the model's predicted probabilities are well-calibrated, we backtest against historical data. For each PD bucket, we compare the model's average predicted default rate to the actual observed default rate.

A well-calibrated model will have a Ratio close to 1.0 across all buckets. Values significantly above 1.0 indicate under-prediction (model is too optimistic); below 1.0 indicates over-prediction (model is too conservative).

7. Deployment

The application is deployed as a containerised service:
  • Backend: A FastAPI server loads the unified XGBoost model (raw + calibrated) and the full company index into memory at startup. All 12-month PDs are pre-computed for instant peer comparison. A single SHAP explainer handles both horizons.
  • Frontend: A static HTML/JS interface served by the same container, consuming the backend API for predictions, SHAP explanations, peer comparisons, and calibration data. CDS spread analysis runs entirely client-side for instant feedback.
  • Data: Large data files (Parquets) are managed via Git LFS to ensure reproducibility.
